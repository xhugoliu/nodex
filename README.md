# Nodex

> 面向节点拓展的 AI 脑图工作台

Nodex 想做的，不是“又一个带聊天框的画布工具”，而是一款把脑图重新变成思考主界面的桌面应用。

它保留熟悉的脑图体验，但把“节点拓展”变成核心交互:

- 一个节点可以继续展开成多个子主题
- 一个节点可以从不同视角继续探索
- 一个节点可以延展成任务、问题、证据或提纲
- 一张图可以在不整体推倒重来的前提下持续重组

当前仓库仍处于产品定义阶段，还不是可直接运行的版本。这个 README 主要用于沉淀产品方向、MVP 范围和技术判断。

## Nodex 想解决什么问题

传统脑图很适合把想法画出来，但不擅长把一个想法持续做深。

常见问题包括:

- 脑图常常停留在“结构草稿”阶段，难以继续推演
- 聊天式 AI 很容易给出内容，但会破坏原有结构
- 内容一旦变多，就很难追踪哪些结论来自哪里
- 修改整张图的成本很高，局部演进体验很差

Nodex 的目标，是让脑图不只是“表达结果”的工具，而是“推进思考”的工作台。

## 产品定位

Nodex 位于三类产品的交叉点:

- 脑图
- 结构化知识工作
- AI 辅助的想法拓展

它更关心这些事情:

- 脑图仍然是主要交互界面
- 节点拓展是标志性能力
- 本地优先存储是默认前提
- AI 修改以结构化 patch 形式发生
- 来源、历史和恢复路径必须清晰

## Nodex 不是什么

Nodex 不希望变成:

> 一个自动生成脑图的工具

它更希望成为:

> 一个以脑图为工作界面，让每个节点都能继续长成结构、证据和行动的工作台

## 核心交互

### 脑图优先

主要工作界面仍然是一张脑图。用户应该可以:

- 从空白脑图开始
- 把文档导入成脑图
- 在画布和大纲之间切换
- 始终保持对整体结构的全局视角

### 节点拓展

每一个节点都可以成为继续思考的起点。典型动作包括:

- `Expand`: 把一个节点拆成更细的子节点
- `Explore`: 按视角、角色或角度继续展开节点
- `Extend`: 把节点延展成任务、时间线、论证链或问题树
- `Compress`: 压缩和总结过于发散的分支
- `Reframe`: 用另一种结构重新组织分支
- `Evidence`: 为节点补充证据、出处和来源

## Patch 驱动的 AI 编辑

AI 不应该直接替换整张图，而应该返回结构化操作，例如:

- 新增节点
- 更新节点
- 移动节点
- 合并节点
- 拆分节点
- 关联来源
- 调整子节点顺序

这样做有几个直接好处:

- 修改可预览
- 修改可撤销
- 修改可检查
- 更适合真实工作场景中的局部编辑

## 适合谁

Nodex 面向的不只是想“画一张图”的人，而是那些需要把想法持续做深的人:

- 整理资料的研究者
- 把想法整理成提纲的写作者
- 拆解决策和方案的产品建设者
- 构建主题结构的学生
- 想把模糊思路变成可持续知识结构的任何人

## 典型使用场景

### 从一个想法开始

先创建一个根节点，例如:

- `AI 代码评审`
- `小红书增长`
- `新能源供应链`

然后继续展开为:

- 定义
- 参与方
- 风险
- 机会
- 下一步

### 从资料开始

导入 PDF、Markdown 或 TXT 文件，生成一张初始脑图，其中包含:

- 一个根主题
- 一级分支
- 带来源关联的节点
- 后续可继续拓展的建议

### 拓展某个具体节点

选中一个节点后，可以继续按不同目标展开:

- 按原因展开
- 按问题展开
- 按风险展开
- 按行动展开
- 按证据展开

### 从结构生成输出

基于当前脑图或当前分支生成:

- Markdown 大纲
- 会议议程
- 任务清单
- 发言提纲
- 研究总结

## MVP 范围

第一版先把这几件事做好:

1. 创建和编辑空白脑图
2. 导入 PDF / Markdown / TXT
3. 从资料生成初始脑图
4. 用 AI 拓展选中的节点
5. 预览并应用结构化 AI patch
6. 关联并查看来源
7. 保存快照并恢复版本
8. 导出 Markdown 大纲

## 固定技术栈

当前唯一明确的技术选择只有:

- `Tauri`
- `LangChain`
- `SQLite`

其他部分暂时保持灵活:

- 前端框架
- LangChain 运行时语言
- 画布实现方式
- 第一版是否需要 LangGraph

## 建议架构

```text
+--------------------------+         +----------------------------+
|  Tauri 前端              | <-----> |  Tauri 后端                |
|  画布 / 大纲 / 检查器 /  | invoke  |  项目 / 文件 / db /        |
|  AI 操作                 | events  |  快照 / patch 应用         |
+--------------------------+         +--------------+-------------+
                                                   |
                                                   |
                                                   v
                                     +----------------------------+
                                     | LangChain 运行时           |
                                     | 导入 / 拓展 / 问答         |
                                     | (Python or JS)             |
                                     +--------------+-------------+
                                                   |
                           +-----------------------+-----------------------+
                           |                                               |
                           v                                               v
                +--------------------------+                   +------------------------+
                | 模型提供方               |                   | 本地项目存储           |
                | 云端或本地               |                   | SQLite + 文件          |
                +--------------------------+                   +------------------------+
```

## 本地优先的数据模型

每个项目应当包含:

- 当前脑图状态
- 来源文件
- 来源切片
- AI 修改历史
- 快照
- 导出结果

建议的存储策略:

- 使用 `SQLite` 存储节点、边、历史、搜索和元数据
- 使用本地文件存储原始导入、快照和导出文件

## 路线图

### 第一阶段

- 项目初始化
- 空白脑图编辑
- SQLite schema
- 基础 patch 应用

### 第二阶段

- 文档导入
- 来源切片
- 初始脑图生成

### 第三阶段

- `Expand` / `Explore` / `Extend` 节点动作
- patch 预览
- 快照与恢复

### 第四阶段

- 基于来源的问答
- 导出流程
- 脑图清理与结构重组工具

## 设计原则

- 脑图优先
- 节点拓展是产品标志能力
- 默认本地优先
- AI 编辑必须基于 patch
- 来源必须可追溯
- 版本历史必须可见
- 聊天只是入口，不是产品本体

## 当前状态

Nodex 目前还处于产品定义阶段，但方向已经相对清晰:

- 保留脑图这一核心隐喻
- 让节点拓展成为差异化能力
- 保持技术核心简单且本地优先

接下来更重要的不是一次性把功能铺满，而是先验证以下三件事:

- 脑图作为主界面是否足够高频
- 节点拓展是否真的比聊天更高效
- patch 驱动的 AI 编辑是否能带来更可信的体验
