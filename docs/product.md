# 产品定位

## Nodex 是什么

Nodex 不是“又一个带聊天框的画布工具”，而是一个把脑图重新变成思考主界面的工作台。

它保留脑图这种熟悉的结构表达方式，但把“节点拓展”变成最核心的交互动作：

- 一个节点可以继续展开成多个子主题
- 一个节点可以从不同视角继续探索
- 一个节点可以延展成任务、问题、证据或提纲
- 一张图可以在不整体推倒重来的前提下持续重组

## 想解决的问题

传统脑图适合把想法画出来，但不擅长把一个想法持续做深。

常见断点包括：

- 脑图停留在结构草稿阶段，难以继续推演
- 聊天式 AI 虽然能给内容，但容易破坏原有结构
- 内容一多，就很难追踪结论、证据和修改过程
- 修改整张图的成本高，局部演进体验差

Nodex 的目标是让脑图不只是“表达结果”的工具，而是“推进思考”的工作台。

## 产品边界

Nodex 更接近这三类产品的交叉点：

- 脑图
- 结构化知识工作
- AI 辅助的想法拓展

Nodex 当前不想做成：

> 一个自动生成完整脑图的工具

Nodex 更想成为：

> 一个以脑图为工作界面，让每个节点都能继续长成结构、证据和行动的工作台

这也意味着：

- 当前仓库里的桌面壳不应被误解为最终产品形态
- 它更接近一层为了验证 patch、来源链路和 AI 审计边界而存在的过渡性工作台
- 后续桌面端如果重做，应该保留现有共享内核与 patch-first 边界，而不是重新回到聊天驱动 UI

## 核心交互

### 脑图优先

主要工作界面仍然是一张脑图。用户应该可以：

- 从空白脑图开始
- 把资料导入成脑图
- 在画布和大纲之间切换
- 始终保持对整体结构的全局视角

当前仓库里的桌面壳还没有达到这个目标。

它已经验证了：

- 本地工作区可以接进桌面环境
- patch / source / AI run 审计链路可以被桌面复用

但它还不是一版适合人类高频工作的脑图主界面。

### 节点拓展

每个节点都可以成为继续思考的起点。

典型动作包括：

- `Expand`：把一个节点拆成更细的子节点
- `Explore`：按视角、角色或角度继续展开节点
- `Extend`：把节点延展成任务、时间线、论证链或问题树
- `Compress`：压缩和总结过于发散的分支
- `Reframe`：用另一种结构重新组织分支
- `Evidence`：为节点补充证据、出处和来源

## Patch-first 原则

Nodex 的 AI 编辑不应该直接替换整张图，而应该返回结构化操作。

这样做的直接收益是：

- 修改可预览
- 修改可撤销
- 修改可审查
- 更适合真实工作场景中的局部编辑

这也是为什么当前 CLI MVP 先落的是 patch 引擎，而不是聊天界面。

## AI Runtime 边界

Nodex 可以试验不同 AI runtime，但 runtime 不是产品本体。

当前已经开始做 LangChain 最小试点，不过它的角色仍然受同一条边界约束：

- LangChain 可以负责模型调用与结构化输出编排
- LangChain 不接管本地状态
- LangChain 不替代 canonical patch
- LangChain 不替代本地历史与审计

换句话说，Nodex 可以让 LangChain 成为“怎么得到 patch”的一层，
但不会让它变成“系统最终怎么改状态”的一层。

## 适合谁

Nodex 面向的不是只想“画一张图”的人，而是那些需要把想法持续做深的人：

- 整理资料的研究者
- 把想法整理成提纲的写作者
- 拆解决策和方案的产品建设者
- 构建主题结构的学生
- 想把模糊思路变成可持续知识结构的任何人

## 设计原则

- 脑图优先
- 节点拓展是标志性能力
- 默认本地优先
- AI 编辑必须基于 patch
- 来源必须可追溯
- 版本历史必须可见
- 聊天只是入口，不是产品本体
